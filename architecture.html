System Architecture

Splitwise Project Architecture

This visualization summarizes how the Splitwise project is structured across the browser, frontend SPA, backend API, business-service layer, and MongoDB persistence layer. The application follows a straightforward client-server model with invitation and activity tracking as the main orchestration mechanisms.

React + Vite Single-page frontend application
Spring Boot REST API and business logic
MongoDB Primary document database
JWT Auth Stateless API protection
Frontend/UI API/Backend Business Services Data/Persistence Cross-cutting/Support

1. End-to-End System View

The browser loads a static React application, which communicates with the Spring Boot API over JSON HTTP requests. The backend validates JWTs, executes domain workflows, and persists core documents in MongoDB.

User Browser

Interacts with the app, submits forms, views groups, expenses, invitations, and activity.

Frontend SPA

React 19 + Vite app with one main UI shell in frontend/src/App.tsx.

Local Storage

Stores authToken, currentUserId, and local settle-state hints.

REST Controllers

Auth, Users, Groups, Expenses, Activities, and Notifications endpoints under /api.

JWT Filter

Validates Bearer tokens for protected requests before controller execution.

Business Services

UserService, GroupService, ExpenseService, NotificationService, EmailService, JwtService.

Activity & Invitation Logic

Acts as the coordination layer for invites, owed amounts, and settlement history.

MongoDB

Stores users, groups, expenses, activities, and pending invitations as document collections.

2. Frontend Architecture

Bootstrap Layer

  • frontend/src/main.tsx mounts the React app.
  • frontend/src/App.tsx holds nearly all runtime UI state.
  • vite.config.ts keeps the build setup minimal.

Client Responsibilities

  • Manage auth token and current user state.
  • Fetch users, groups, expenses, activities, and invitations.
  • Compute UI-only values like user share, totals, and charts.
  • Render tabs for groups, expenses, friends, activity, and account views.

API Integration

  • Uses authedFetch to add Bearer tokens.
  • Reads API base URL from VITE_API_BASE_URL.
  • Handles 401 by clearing local session state.

4. Core Domain Model

Primary Documents

  • User: identity, email, password hash, friend IDs, notification preference
  • Group: name, owner ID, member IDs
  • Expense: payer, participants, amount, group reference, custom splits, bill URL
  • Activity: user-facing event log for owed/settled/group/friend actions
  • PendingInvitation: friend or group invitation, keyed by email and/or user ID

Relationship Summary

  • Users have mutual friendships through friendIds.
  • Groups are owned by a user and contain member user IDs.
  • Expenses belong either to a user directly or to a group.
  • Activities are per-user timeline entries rather than shared events.
  • Pending invitations bridge onboarding and membership acceptance flows.

5. Key Runtime Flows

Authentication Flow

1
User submits signup or login from the frontend.
2
/api/auth validates credentials or creates the user.
3
Backend hashes passwords with BCrypt and returns a JWT token.
4
Frontend stores the token and sends it on future API requests.

Friend Invitation Flow

1
User adds a friend by email from the frontend.
2
UserService creates a PendingInvitation.
3
If the invited person later signs up, the invitation is linked to their user ID.
4
On acceptance, both users are updated and activity entries are written.

Group Creation Flow

1
Frontend submits group name, owner, and selected members.
2
GroupService saves the group with only the owner as an immediate member.
3
Other selected users receive group invitations instead of direct membership.
4
Accepting an invite updates the group and appends activities for both sides.

Expense & Settlement Flow

1
Frontend builds an expense payload, including custom or percentage splits.
2
ExpenseService normalizes defaults and validates split totals.
3
Expense is stored in MongoDB and owed activities are created for participants.
4
Settlement writes corresponding payer and settler activity records.

6. Deployment View

Frontend Deployment

  • Built with Vite in the frontend Docker image.
  • Static files copied into Nginx.
  • API base URL baked in via build arg and environment variable.

Backend Deployment

  • Built as a Spring Boot fat JAR with Maven.
  • Runs in a Java 17 runtime container.
  • Configured through MONGODB_URI, JWT_SECRET, and PORT.

Persistent Storage

  • MongoDB holds the operational data model.
  • No cache, queue, or secondary analytics database is present.
  • Activity and invitation collections act as lightweight workflow state.

7. Architecture Notes

Strengths

  • Clear backend layering with thin repositories.
  • Simple and understandable document model.
  • Invitation-first design supports both existing users and email onboarding.
  • Activity logs provide a built-in user timeline and operational audit trail.

Current Constraints

  • The frontend is concentrated in one large component, which mixes view and data logic.
  • JWT validation exists, but authorization is still largely parameter-driven.
  • Settlement state is partly local UI state and partly backend activity history.
  • Email sending is currently a development stub rather than a real integration.
